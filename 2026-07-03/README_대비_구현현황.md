## 1. 화면(모듈) 10개 대비

### 모듈 1~3, 7, 9, 10 (스플래시/온보딩/메인/스트레칭/알림설정/마이페이지)
기획안 항목을 거의 그대로 구현했다. 다만:
- **스트레칭 모듈**: "동작별 이미지"라고 되어 있지만 실제로는 **이모지 텍스트**를 쓴다(`item_stretching.xml`의 `tvStretchEmoji`). 실제 사진/일러스트 에셋을 준비하지 않고 프로젝트를 진행했기 때문에, 앱 전체에서 이미지 대신 이모지를 쓰는 방식으로 통일했다.
- **마이페이지 누적 통계**: "키운 캐릭터 수"라고 되어 있지만, 계정 하나당 펫이 **한 마리만** 있는 구조라 "캐릭터를 몇 마리 키웠는지" 셀 대상이 없다. 대신 지금 키우고 있는 **펫의 레벨**(`🐤 키운 펫 레벨: Lv.%d`)을 보여준다.

### 모듈 4: DrinkHistoryActivity
기획안 그대로: 오늘 기록 목록, 삭제/수정, 남은 양, 달력으로 날짜별 달성 여부(파란색: 달성, 회색: 기록 있으나 미달성) 표시. 색만 "초록/회색" 대신 앱 전체 팔레트에 맞춰 "파란색/회색"으로 바뀌었을 뿐 기능은 기획과 동일하다.

### 모듈 5: StatisticsActivity
- 일간 섭취량은 기획대로 **막대 그래프**.
- 주간은 기획에 "꺾은선(라인) 그래프"라고 적혀 있지만 실제로는 **막대 그래프**로 구현했다. MPAndroidChart 없이 Canvas로 직접 그리는 `BarChartView` 하나만 만들고 일간/주간 둘 다 재사용했기 때문이다(라인 차트를 별도로 구현하지 않음). 월간 요약·지난달 대비 비교는 기획대로 구현.

### 모듈 6: CharacterActivity — 가장 크게 바뀐 부분
기획: "아이템 상점에서 모자/액세서리/배경 아이템을 사서 펫에 장착".

처음에는 기획대로 모자·선글라스·리본·목도리 등을 이모지로 펫 위에 겹쳐서 보여주는 방식으로 구현했다. 그런데 **펫 종류(병아리/강아지/고양이)마다, 심지어 같은 종이어도 성장 단계마다 이모지 그림의 구도가 완전히 달라서**(정면 얼굴 클로즈업 단계도 있고, 옆모습 전신 그림 단계도 있음) 액세서리 위치를 아무리 좌표로 맞춰도 계속 눈·머리 위치가 어긋나는 문제가 있었다. 여러 차례(종×성장단계×화면크기 조합을 에뮬레이터로 하나하나 확인) 튜닝을 시도했지만 근본적으로 유니코드 이모지는 위치 정보가 없는 완성된 그림이라 한계가 있었다.

그래서 사용자와 상의 후 **모자/액세서리 개념을 없애고, 펫 뒤 원형 배경을 단색으로 바꾸는 방식**(하늘색/코랄/민트/레몬/라벤더 5종)으로 단순화했다. `items` 테이블 구조(가격/보유/장착 여부)는 그대로 재사용하되, `type`은 `background` 하나만 쓰고 `emoji` 컬럼에는 실제 이모지 대신 **색상 hex 코드**를 저장한다. 펫 도감(성장 단계 컬렉션)과 레벨/경험치/기분 표시는 기획대로 구현되어 있다.

### 모듈 8: RankingActivity
기획: "친구 추가(닉네임 검색) + 친구 목록 + 주간 랭킹 1위 보상".

주간 랭킹 1위 보상은 기획대로 구현했지만, **닉네임 검색으로 친구를 추가하는 기능은 구현하지 않았다**. 서버가 없는 로컬 전용 앱이라 실제로 검색할 대상(다른 사용자 서버 데이터)이 없기 때문에, 대신 이 기기에 저장된 로컬 계정들과 미리 만들어둔 목업 친구 데이터(`RankingCatalog`)를 섞어서 랭킹처럼 보여주는 것으로 대체했다. `RankingCatalog`에 이름으로 랭킹을 만들어주는 `friendFromNickname()` 함수는 만들어뒀지만 실제 "친구 추가" UI가 없어서 아직 어디서도 호출되지 않는 죽은 코드로 남아 있다.

## 2. 계획에 없던 추가 화면: LoginActivity

기획안 10개 모듈에는 로그인/계정 화면이 없다. 서버 인증이 없는 로컬 전용 앱인데, 개발/테스트 중 **한 기기에서 여러 계정을 만들어 써야 할 일이 많아서**(펫 종류별 확인, 기능 테스트 등) 로컬 멀티 프로필 개념을 추가했다. 온보딩을 마치면 계정이 하나 만들어지고, 로그아웃하면 이 기기에 저장된 계정 중 하나를 골라 다시 들어가는 화면(`LoginActivity`)이다. 실제 서버 로그인이 아니라 계정을 탭하기만 하면 바로 들어가는 방식이다.

## 3. 기술 스택 표 대비

| 기획 | 실제 | 이유 |
|---|---|---|
| Java | Kotlin | 처음부터 Kotlin으로 시작함 (README 초안 작성 후 언어를 바꿈) |
| API 26 | API 29 | 개발 중 사용한 최신 API/라이브러리 호환을 위해 상향 |
| MPAndroidChart | 자체 BarChartView | 외부 차트 라이브러리 의존성 없이 Canvas로 직접 그림 |
| Glide | 미사용 | 사진 에셋이 없고 전부 이모지/색상으로 표현하기 때문에 이미지 로더가 필요 없음 |
| Lottie | 미사용 | 캐릭터 애니메이션을 이모지 성장 단계 전환으로 대체 |
| ViewPager2 | 미사용 | 화면 전환을 BottomNavigationView + Intent 방식으로만 구현, 스와이프 탭 UI가 필요한 곳이 없었음 |
| AlarmManager + NotificationCompat | 동일하게 구현 | 그대로 유지 |
| RecyclerView / BottomNavigationView / CardView | 동일하게 구현 | 그대로 유지 |

## 4. DB 설계 대비

- **users**: 기획과 동일.
- **drink_records**: 기획 항목 + `userId` 추가. 계정이 여러 개일 수 있는 구조(로그인 화면)라 각 기록이 어느 계정 것인지 구분해야 해서 추가됨.
- **characters**: 기획엔 `is_active`(현재 사용 중 여부) 컬럼이 있었지만 실제로는 없다 — 계정 1개당 펫이 항상 1마리뿐이라 "여러 캐릭터 중 활성 캐릭터 고르기" 개념 자체가 필요 없어졌다. 대신 `userId`(소유 계정)와 `species`(병아리/강아지/고양이 종류 — 온보딩에서 고르는 항목)가 추가됐다.
- **items**: 기획 항목은 대부분 유지되지만 `type`은 이제 `background` 하나만 쓰고, `emoji` 컬럼은 색상 hex 문자열을 담는 용도로 바뀌었다(2번 항목 참고). `userId`도 추가(계정별 보유/장착 상태 분리).

## 5. 결론

10개 모듈 모두 화면 자체는 존재하고 정상 작동한다. 기획과 다르게 구현된 것은 크게 세 가지다.

1. **캐릭터 꾸미기**: 모자/액세서리 → 배경 색상으로 단순화 (이모지 좌표 정렬 한계 때문)
2. **랭킹 친구 추가**: 미구현 (서버 없는 로컬 앱 구조상 검색 대상 부재, 로컬 계정+목업 데이터로 대체)
3. **통계 주간 그래프**: 꺾은선 → 막대그래프 (자체 제작 차트 하나로 통일)

그 외 기술 스택 차이(Java→Kotlin, MPAndroidChart/Glide/Lottie/ViewPager2 미사용)는 실제 구현 과정에서 필요가 없어져 자연스럽게 빠진 것들이다.
